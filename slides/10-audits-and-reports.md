# Use case 5 — Audits & reports

There is no separate "audit" workflow. **An audit is just a spec** whose deliverable is a report (markdown, CSV, JSON) instead of code.

Same template, same lifecycle, same folder.

Note:
The discipline of capturing audits as specs (committed files in git) is what makes them useful. Chat scrolls away. A committed spec with a linked report is something you can re-open in 3 months, link from a PR, or hand to a contractor.

----

## Real example — `SPEC-001 Twilio Spend Audit`

Already in the repo at `phoenix-docs/specs/active/twilio-spend-audit.md`:

```yaml
---
id: SPEC-001
title: Twilio Spend Audit
status: ready
created: 2026-03-01
author: Anthony Coffey
affected_repos: [phoenix]
---
```

It's a regular feature spec. The `## Overview` describes the audit goal. The `## Requirements` define what data to collect. The `## Design` covers how (Twilio API + Postgres). The `## Tasks` are the atomic steps.

> The audit *output* (the actual report) is a separate file generated by running the spec — typically committed alongside it or to a sibling location chosen in the Design section.

## Audit prompt library

Five reusable read-only audit prompts. Each one fits in `## Design` of an audit spec or runs ad-hoc in plan mode.

```
Security: Read every file in [repo]. Audit for OWASP Top 10
issues. For each finding output: severity, file:line,
description, recommendation, effort (S/M/L). Do not edit.

Dependencies: For [repo], list every dependency with version,
flag CVEs, flag unmaintained (>12mo no commits), flag major
upgrades available with breaking-change risk.

Dead code: Scan [repo] for: exported functions never imported,
unreferenced files, TODO/FIXME/HACK older than 6 months
(use git blame), commented-out blocks >5 lines.

Coverage: For [repo] list every public function in
src/services/. For each, mark covered / partial / uncovered.
Prioritize by criticality.

Convention: Read [repo]/CLAUDE.md and phoenix-docs/
documentation/development-standards.md. Find files that
violate naming, structure, or import-order conventions.
```

----

## The reusable audit prompt structure

```
TASK: Audit [scope] for [criteria]

CONSTRAINTS:
- Read-only — do not edit any code
- Output as markdown to [path chosen in spec Design section]
- Include severity, file:line, description, recommendation
- Sort by severity (highest first)
- End with a summary count by severity

DO NOT:
- Implement fixes
- Modify any files outside the report itself
```

Use plan mode for audits. The constraints above are belt-and-suspenders — plan mode already prevents edits, but stating it explicitly keeps the report focused on findings rather than fixes.
